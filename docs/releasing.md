# Releasing Lios

GitHub Actions is the only producer of official release artifacts. Files created under a
developer's local `target/` directory are smoke-test outputs and must not be uploaded as a
formal release.

## Version source of truth

The authoritative version lives in the root `Cargo.toml`:

```toml
[workspace.package]
version = "0.1.0"
```

Every Rust workspace package uses `version.workspace = true`. `Cargo.lock`, `package.json`,
`package-lock.json`, and `src-tauri/tauri.conf.json` are synchronized mirrors for their build
tools.

```bash
npm run version:sync
npm run version:check
```

The release workflow also requires the pushed tag to equal `v` plus the workspace version.

## Release matrix

One semantic version tag creates one GitHub Release. Asset prefixes keep desktop and CLI
deliverables distinct.

Current desktop assets:

```text
lios-desktop-vX.Y.Z-windows-x86_64-nsis.exe
lios-desktop-vX.Y.Z-linux-x86_64.deb
lios-desktop-vX.Y.Z-linux-x86_64.rpm
lios-desktop-vX.Y.Z-linux-x86_64.AppImage
lios-release-vX.Y.Z-rpm-signing-public.asc
```

The future headless CLI matrix is reserved as:

```text
lios-cli-vX.Y.Z-linux-x86_64.tar.gz
lios-cli-vX.Y.Z-linux-x86_64.deb
lios-cli-vX.Y.Z-linux-x86_64.rpm
lios-cli-vX.Y.Z-linux-aarch64.tar.gz
lios-cli-vX.Y.Z-linux-aarch64.deb
lios-cli-vX.Y.Z-linux-aarch64.rpm
```

The workflow intentionally refuses to release after a `lios-cli` package appears until the
signed x86_64 and ARM64 jobs are implemented. Windows does not receive a standalone CLI asset:
the current-user desktop installer adds its installation directory to PATH, so the same signed
`lios.exe` can serve both the desktop and future CLI modes. A new terminal is required to observe
the updated PATH.

## Signing policy

Every prerelease and stable release is fully signed. Missing signing material fails the
workflow before GitHub Release creation.

- Windows NSIS, its installed `lios.exe`, and its uninstaller: Authenticode with an RFC 3161
  timestamp. The bundled Microsoft `WebView2Loader.dll` must also retain a valid Authenticode
  signature.
- RPM: native OpenPGP package signature generated by Tauri. The expected primary-key fingerprint
  is a reviewed repository trust anchor, and the matching public key is published as a
  Sigstore-signed asset.
- Every payload and `SHA256SUMS`: a keyless Sigstore bundle (`*.sigstore.json`).
- Every uploaded asset: GitHub artifact attestation generated by `actions/attest`.

Required secrets in the GitHub `release` Environment:

```text
WINDOWS_CERTIFICATE_BASE64
WINDOWS_CERTIFICATE_PASSWORD
RPM_SIGNING_KEY
RPM_SIGNING_KEY_PASSPHRASE   # optional when the private key is unencrypted
```

Windows signatures use the RFC 3161 endpoint `http://timestamp.digicert.com`. The PFX must
contain a publicly trusted code-signing certificate and private key. `RPM_SIGNING_KEY` must
contain the complete ASCII-armored private key. Before the first release, replace `UNCONFIGURED`
in `.github/release/rpm-signing-key.fingerprint` with its full primary-key fingerprint. Rotating
the RPM key requires a reviewed change to that committed trust anchor.

Sigstore and GitHub attestations use GitHub Actions OIDC, so they do not require a stored
private key. Artifact attestations on private repositories require an eligible GitHub
Enterprise Cloud plan; on Free, Pro, and Team plans they are available for public
repositories.

## Cutting a release

Repository setup is part of the release boundary:

- Protect the `v*` tag namespace with a GitHub tag ruleset or protection rule.
- Create a `release` Environment with required reviewers and store the signing secrets listed
  above there. The workflow runs source verification before requesting approval, then blocks both
  platform-signing jobs on that approval.
- Keep the default branch protected. A release tag is rejected unless its commit is already an
  ancestor of the repository's default branch.

1. Change `[workspace.package].version` in the root `Cargo.toml`.
2. Run `npm run version:sync`.
3. Run the full local verification suite and commit the version change.
4. Create and push the matching tag, for example `v0.2.0`.
5. The `Signed release` workflow tests, builds, signs, verifies, attests, and finally creates
   the GitHub Release.

The Windows job performs a silent current-user install and uninstall before publishing. That
smoke test verifies the release-certificate thumbprint on the installer, installed executable,
and uninstaller; requires their timestamp certificates; validates the WebView2 loader signature;
and confirms PATH is added and removed. Before publication, the workflow also confirms that the
release tag still resolves to the exact commit that triggered the run.

Security-sensitive third-party Actions are pinned to full commit SHAs. Cosign signing, GitHub
attestation, and GitHub Release publication run as separate jobs so no job combines OIDC signing
authority with `contents: write`.

Linux packages are built on Ubuntu 22.04 for an older glibc baseline. Native RPM verification runs
with a digest-pinned Fedora 42 container because Ubuntu 22.04's older RPM 4.17 cannot validate the
OpenPGP signature format emitted by the current Tauri bundler.

The workflow never creates an unsigned draft or partially signed release. A failed job leaves
no GitHub Release behind.

## Consumer verification

Verify checksums and Sigstore bundles:

```bash
sha256sum --check SHA256SUMS
cosign verify-blob \
  --bundle ASSET.sigstore.json \
  --certificate-identity "https://github.com/Lingbou/Lios/.github/workflows/release.yml@refs/tags/vX.Y.Z" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
  ASSET
```

Verify GitHub provenance:

```bash
gh attestation verify ASSET -R Lingbou/Lios
```

Windows users can inspect Authenticode with `Get-AuthenticodeSignature`. RPM users should first
verify `lios-release-vX.Y.Z-rpm-signing-public.asc` with its Sigstore bundle, import that verified
key, and then check the package:

```bash
rpm --import lios-release-vX.Y.Z-rpm-signing-public.asc
rpm -Kv lios-desktop-vX.Y.Z-linux-x86_64.rpm
```
